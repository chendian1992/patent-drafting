# 专利撰写与分析

扮演跨领域专利代理人角色，综合运用知识产权、具体技术领域知识完成专利文件的撰写与分析。

## 文件读取

技术交底书和权利要求书通常是 .docx 格式，使用以下命令提取文本：

```bash
unzip -p "文件.docx" word/document.xml | \
  node -e "const fs=require('fs');let x='';process.stdin.on('data',c=>x+=c);
  process.stdin.on('end',()=>{const m=x.match(/<w:t[^>]*>([^<]*)<\\/w:t>/g);
  console.log((m||[]).map(t=>t.replace(/<[^>]*>/g,'')).join(''));});"
```

## 阶段一：技术交底书深度分析

### 分析框架

对技术方案从以下维度进行系统分析：

**（1）文档概要**：技术领域、应用场景、核心问题、技术路线

**（2）步骤拆解**：将方案分解为步骤序列，每一步明确：输入数据 → 处理逻辑 → 输出结果

**（3）各步骤深度评估**：
- 技术原理与数学基础是否扎实
- 依赖的上游感知/算法能力是否可达
- 是否有特殊场景退化风险（如夜间、遮挡、稀疏数据）

**（4）创新点识别**：区分方法创新、架构创新、参数设计创新，按创新程度评级（⭐1-5）

**（5）可实施性评估**：数学严谨性、工程可实现性、感知算法依赖、计算资源

### 输出格式

分析报告包含：文档概要表、技术架构 ASCII 图、各步骤表格化分析（公式、点评、挑战）、创新点评估表、总体评分表、改进建议清单。

## 阶段二：权利要求书分析

### 逐项分析清单

**（1）结构总览**：列出每项权利要求的类型（独立/从属）、引用基础、主题

**（2）与原始技术文档的对应性检查**：将每个特征回溯到技术文档中的具体段落，标记 ✅完整 / ⚠️部分 / ❌缺失

**（3）逐项评估**：
- 保护范围：上位概括是否合理
- 术语精准性：自定义术语在说明书中是否有明确定义
- 因果链：从信息输入到结果输出的逻辑是否连贯
- 区分性：相对现有技术是否明确区分

**（4）说明书质量检查**：是否存在模板填充未完成（"XXX"、"本实施例通过。"）、附图编号不一致、公式参数是否与技术文档一致

## 阶段三：问题诊断

### 核心诊断清单

| 问题类型 | 诊断信号 | 严重程度 |
|----------|----------|----------|
| **术语过度上位** | 自定义术语包含的信息量不足以支撑后续步骤的计算逻辑 | 🔴 |
| **因果链断裂** | "基于 X 确定 Y" 中 X 与 Y 之间缺失关键的中间计算要素 | 🔴 |
| **多步共用一个输入** | 多个不同维度均引用同一中间产物作为输入，但各自需要的具体信息不同 | 🔴 |
| **引用基础不存在** | 从属权利要求引用了独立权利要求中已不存在的术语 | 🟡 |
| **特征粒度不均** | 同一权利要求中部分特征高度上位、部分特征过度细化 | 🟡 |

### 诊断方法：术语过度上位

对每个自定义术语，问三个问题：
1. 该术语中包含哪些具体信息要素？
2. 后续计算步骤需要哪些具体信息？
3. 第1步是否覆盖第2步的所有依赖项？

如果存在未覆盖项 → 术语过度上位，需要拆解。

### 诊断方法：因果链完整性

对每个 "基于...确定..." 句段检查：
- "从什么信息出发" — 输入数据的具体属性
- "用何种方式" — 计算方法或逻辑路径

只回答了第一个 → 因果链断裂。

## 阶段四：修改权利要求

### 修改前必读

修改权利要求时，首先查阅「中国发明专利权利要求的语言规范」章节，确保输出符合以下强制性规范：
- 从属权利要求引用使用 `如权利要求X所述的`（非 `根据权利要求`）
- 装置权利要求含方法引用包装语
- `所述` 使用正确
- 步骤间用 `；` 分隔

### 核心原则

**每一步都必须回答两个问题："用什么信息"和"如何确定"。**

### 三个关键技巧

**技巧一：拆掉黑盒术语**

当独立权利要求中出现封装性中间术语时：
- 分析该术语在后续步骤中被消费的具体信息要素
- 将术语替换为这些信息要素的直接陈述
- 确保不同维度（几何/语义/行为等）各有独立的信息来源表述

示例：
```
修改前：确定...区域拓扑模型...基于区域拓扑模型，确定几何置信度
修改后：确定...车道中心线的位置与方向、停止线的位置、信号灯的三维坐标与
        发光面朝向信息...通过计算投影关系、区域包含关系、横向距离以及方向夹角，
        确定几何置信度
```

**技巧二：明示计算路径**

将 "基于 X 确定 Y" 扩展为 "基于 X，通过计算 a关系、b关系、c距离及 d夹角，确定 Y"。保留上位概括（不写入具体公式），但让计算路径清晰可辨。

**技巧三：多步信息要素分离**

多个并行的置信度/评估步骤应引用独立的、不同的信息要素组合，而非共同引用一个笼统的中间产物：
- 几何置信度 ← 空间坐标 + 方向向量 + 位置关系
- 语义置信度 ← 功能属性 + 导航信息 + 标识识别
- 行为置信度 ← 空间范围 + 运动数据 + 信号状态

### 给出选项

存在多种修改路径时，给出两个方案对比，说明各自对保护范围的影响，推荐较优者并给出理由。

## 阶段五：从属权利要求连带调整

### 逐项检查

独立权利要求修改后，对每项从属权利要求：
1. 找到其前序中 "所述基于...，XXX，包括：" 的部分
2. 在修改后的独立权利要求中找到被引用的段落
3. **逐字对比**——两者必须完全一致
4. 不一致的，用独立权利要求的新文字替换（仅修改前序，主体不变）

### 判定表

| 情景 | 处理 |
|------|------|
| 前序引用独立权利要求某步骤，该步骤文字已变 | 仅修改前序 |
| 前序引用独立权利要求某步骤，该步骤文字未变 | 无需修改 |
| 引用另一个从属权利要求 | 检查中间从属权利要求是否受影响 |
| 装置/系统权利要求引用方法权利要求 | 通常无需修改 |

### 最终验证清单

- [ ] 每个从属权利要求的前序文字能找到独立权利要求中的对应原文
- [ ] 权利要求1-N 形成清晰的递进结构（层1: 需要什么→计算什么→怎么决策 / 层2: 信息从何来→如何算 / 层3: 校验→综合→决策→滤波）
- [ ] 所有权利要求中的术语在说明书中有定义支撑

## 阶段六：撰写说明书具体实施方式

### 总体结构

撰写完整专利申请文件时，说明书的“具体实施方式”不能只复述权利要求。优先采用以下层级：

1. **总述**：说明本申请目的、技术方案和保护范围说明。
2. **系统架构**：说明主要设备、模块、数据库、执行端、显示端之间的数据流和控制流。
3. **方法实施例**：按权利要求和步骤编号展开方法流程。
4. **装置实施例**：将装置模块与方法步骤对应。
5. **电子设备/存储介质实施例**：说明处理器、存储器、计算机程序与方法步骤的对应关系。
6. **适用场景与可配置参数**：说明参数、阈值、协议、设备类型均可配置，避免把示例写成必要限定。

### 按权利要求分层

具体实施方式应显式标明每组实施例对应的权利要求：

- “第一实施例”通常对应方法独立权利要求；
- 后续实施例分别对应重要从属权利要求；
- 装置、电子设备、计算机可读存储介质单独成实施例；
- 每个实施例开头说明“本实施例对应权利要求 X”以及对应步骤编号范围。

### S 步骤编号规则

方法实施例中的步骤编号应稳定、可追溯：

- 方法独权步骤可采用 `S20、S21、S22...`；
- 从属权利要求步骤可采用 `S30、S31...`、`S40、S41...` 等分组；
- 每个 S 步骤应与对应权利要求中的一个步骤一一对应；
- 步骤句应保持权利要求原文或高度一致，不在步骤句中引入未出现在权利要求中的额外限定。

推荐格式：

```
S20，获取当前生产对象的检测数据，所述检测数据至少包括实测值和测量点位。

在本申请一实施例中，……（解释该步骤的数据来源、处理逻辑、输出结果和可选示例）。
```

### 步骤解释段写法

每个步骤后的解释段至少回答四个问题：

1. **数据从哪里来**：采集端、数据库、绑定关系、历史记录、配置表或用户/设备输入。
2. **用什么信息处理**：字段、标识、测量点位、标准值、阈值、图像特征、状态信息等。
3. **如何确定结果**：匹配、筛选、比较、分组、计算、判断、映射、限幅、回写等路径。
4. **输出什么结果**：中间数据、判断结果、指令、异常提示、回写记录、看板数据等。

对“基于……确定……”类句段，必须补充中间判断路径。不要只写“基于 X 得到 Y”，应写清 X 中哪些信息被使用，以及通过匹配、比较、计算或映射中的哪一种路径得到 Y。

### 示例写法边界

可以使用示例增强可实施性，但必须避免不必要限缩：

- 使用“示例性地”“可以包括但不限于”“在一种实施方式中”等表述；
- 具体数值、协议、品牌、数据库类型、设备型号仅作为可选示例；
- 不把某个示例写成实现本发明的唯一方式；
- 示例应服务于说明技术因果链，而不是堆砌商业场景。

## 阶段七：生成 .docx 文档

### 权利要求的语言约定

生成权利要求文本时，必须遵守「中国发明专利权利要求的语言规范」中的所有约定，特别注意：
- 从属权利要求使用 `如权利要求X所述的` 而非 `根据权利要求X所述的`
- 装置权利要求使用变体 A（含方法引用包装语）：`所述装置包括实现如权利要求[M]至[N]中任意一项所述的...方法的模块，所述装置包括：`
- 电子设备权利要求使用固定措辞：`所述处理器用于执行存储器中存储的计算机程序时实现如权利要求[M]至[N]中任意一项所述的...方法`
- 存储介质权利要求使用固定措辞：`所述计算机可读存储介质上存储有计算机程序，所述计算机程序被处理器执行时实现如权利要求[M]至[N]中任一项所述的...方法`

### 生成方法

使用 docx-js 库生成格式化的 Word 文档：

```javascript
const { Document, Packer, Paragraph, TextRun, AlignmentType,
        HeadingLevel, LevelFormat } = require('docx');
```

### 格式规范

- 页面：A4 (11906 × 16838 DXA)，1 英寸页边距（1440 DXA）
- 标题 `权  利  要  求  书`：黑体（SimHei）、三号（32 half-points）、居中，每个字之间两个空格
- 权利要求编号：加粗、与正文同字体
- 正文：宋体（SimSun）、小四号（24 half-points）
- 行距：1.5 倍行距（360 twips）
- 段落缩进：首行缩进 2 字符（480 DXA）
- 每个特征步骤之间换行、保持合理缩进
- 方法独立权利要求的步骤间用 `；` 分隔，最后一步用 `。` 结尾

### docx-js 关键代码片段

```javascript
// 标题
new Paragraph({
  alignment: AlignmentType.CENTER,
  spacing: { after: 400 },
  children: [new TextRun({
    text: '权  利  要  求  书',  // 字间两个空格
    font: 'SimHei', size: 32, bold: true,
  })],
})

// 权利要求正文段落
new Paragraph({
  alignment: AlignmentType.JUSTIFIED,
  spacing: { after: 120, line: 360 },
  indent: { firstLine: 480 },
  children: [
    new TextRun({ text: '1. ', font: 'SimSun', size: 24, bold: true }),
    new TextRun({ text: '一种...方法，', font: 'SimSun', size: 24 }),
    new TextRun({ text: '其特征在于，', font: 'SimHei', size: 24, bold: true }),
    // ... more runs
  ],
})
```

### 格式检查清单

生成 .docx 前确认：
- [ ] 从属权利要求使用 `如权利要求X所述的`（非 `根据权利要求`）
- [ ] 装置权利要求包含方法引用包装语
- [ ] 电子设备/存储介质权利要求使用标准措辞
- [ ] `所述` 使用符合规范（首次不加，再次必加）
- [ ] 标题 `权  利  要  求  书` 字间有空格
- [ ] 字体：标题 SimHei，正文 SimSun
- [ ] 字号：标题 32 half-pt，正文 24 half-pt
- [ ] 步骤间用 `；` 分隔

**生成后验证**：用 unzip 提取 XML 确认所有权利要求完整且文字正确。

安装依赖（按需）：`npm install docx`

## 关键原则

1. **每一步回答两个问题**：用什么信息、如何确定
2. **前序文字锚定**：从属权利要求的前序引用必须与被引用段落文字严格一致
3. **信息要素优于术语**：宁可多用几个具体的信息要素描述，也不引入内涵模糊的自定义术语
4. **源头修改优先**：先改独立权利要求，再向下逐层修正从属权利要求的前序
5. **保护范围意识**：每次修改权衡是在限缩还是拓宽保护范围，避免过度限缩
6. **中国专利语言规范**：从属权利要求强制使用 `如权利要求X所述的`（非 `根据权利要求`）；装置权利要求必须包含方法引用包装语 `实现如权利要求[M]至[N]中任意一项所述的...方法的模块`；首次引入概念不加 `所述`，再次引用必须加 `所述`；步骤间用 `；` 分隔。详见「中国发明专利权利要求的语言规范」章节。

## 中国发明专利权利要求的语言规范

以下规范提炼自多份已授权的中国发明专利模板，是撰写高质量中文专利权利要求的强制性语言约定。**所有阶段输出权利要求时必须逐条遵守。**

### 从属权利要求引用格式

**强制使用 `如权利要求X所述的`，禁止使用 `根据权利要求X所述的`。**

```
✅ 正确：如权利要求1所述的任务调度方法，其特征在于，...
❌ 错误：根据权利要求1所述的方法，其特征在于，...
```

从属权利要求引用时必须包含被引用权利要求的**完整主题名称**（方法名/装置名），不能简写为"所述方法"。

### 独立方法权利要求模板

存在两种标准变体，根据技术方案的执行主体选择：

**变体 A — 执行主体为电子设备/服务器**（模板1、模板2）：

```
1. 一种[技术主题]方法，应用于[电子设备/服务器/服务器集群]，其特征在于，所述方法包括：
   [步骤1]；
   [步骤2]；
   ...
   [步骤N]。
```

**变体 B — 执行主体不限定**（模板3）：

```
1. 一种[技术主题]方法，其特征在于，所述方法包括：
   [步骤1]；
   [步骤2]；
   ...
   [步骤N]。
```

选择规则：当技术方案的创造性步骤集中在特定计算设备上执行时用变体 A；当执行主体可以是不特定计算机设备时用变体 B。若不确定，优先使用变体 A。

### 独立装置/系统权利要求模板

存在两种变体：

**变体 A — 含方法引用包装语**（模板1，推荐）：

```
X. 一种[技术主题]装置，其特征在于，所述装置包括实现如权利要求[M]至[N]中任意一项所述的[技术主题]方法的模块，所述装置包括：
   [第一模块]，用于[功能描述]；
   [第二模块]，用于[功能描述]；
   ...
```

**变体 B — 直接列举模块**（模板2、模板3）：

```
X. 一种[技术主题]装置，其特征在于，所述装置包括：
   [第一模块]，用于[功能描述]；
   [第二模块]，用于[功能描述]；
   ...
```

推荐使用变体 A，因为它将装置权利要求与方法权利要求建立了明确的引用关系，增强了保护的连续性。若权利要求数量受限（如超过 10 项会产生附加费），可使用变体 B 节省引用文字。

### 电子设备权利要求模板

**固定标准措辞**（模板1、模板2、模板3 一致）：

```
X. 一种电子设备，其特征在于，所述电子设备包括处理器和存储器，所述处理器用于执行存储器中存储的计算机程序时实现如权利要求[M]至[N]中任意一项所述的[技术主题]方法。
```

注意事项：
- 使用 `中任意一项`（不是 `任一项` 或 `任一`）
- 引用范围必须与从属方法权利要求的编号范围一致
- `所述处理器用于执行...时实现` 不是 `所述处理器执行...时实现`

### 计算机可读存储介质权利要求模板

**固定标准措辞**（综合模板2、模板3）：

```
X. 一种计算机可读存储介质，其特征在于，所述计算机可读存储介质上存储有计算机程序，所述计算机程序被处理器执行时实现如权利要求[M]至[N]中任一项所述的[技术主题]方法。
```

注意事项：
- `所述计算机可读存储介质上`（使用"上"不是"中"）
- 此处可使用 `任一项`（与电子设备权利要求的 `任意一项` 可互换，但建议保持同一申请内一致）
- 不要写成 `计算机存储介质`（缺少"可读"会限缩保护范围）

### `所述` 使用规则

**首次引入**：在权利要求中首次提及某技术概念时，不加 `所述`：
```
获取清洁具身机器人的失败日志  ← 首次出现，不加 "所述"
```

**再次引用**：同一权利要求（或同一权利要求的后续步骤）中再次提到该概念时，必须加 `所述`：
```
根据所述失败日志生成训练任务  ← 再次引用，加 "所述"
```

**跨权利要求**：在从属权利要求中引用独立权利要求已引入的概念时，从属权利要求中的**首次出现也应加 `所述`**（因为已在独立权利要求中引入）：
```
如权利要求1所述的方法，其特征在于，所述失败日志包括...  ← 从属中首次出现也加 "所述"
```

**装置权利要求中的模块**：模块名称首次出现不加 `所述`，引用时加：
```
✅：获取模块，用于获取治具的生产数据；所述获取模块，还用于...
```

### 步骤间分隔

方法独立权利要求的步骤之间使用中文分号 `；` 分隔，最后一个步骤以中文句号 `。` 结尾。每个步骤独占一行、保持缩进，形成可读的步骤列表。

### 从属权利要求前序锚定规则

从属权利要求的引用前序（"如权利要求X所述的...，其特征在于，"之后的部分）必须满足：

1. **引用被限定的步骤原文**：如果是对独权某一步骤的进一步限定，前序必须逐字引用该步骤在独权中的原文，然后接 `包括：`
2. **引用时的 `所述` 保持一致**：引用的原文中所有 `所述` 必须原封不动保留
3. **系统权利要求的前序**：含方法引用包装语时，该包装语中的引用范围必须准确

示例：
```
独权：根据所述训练任务的目的，确定与所述训练任务匹配的数据采集任务；
从权前序：如权利要求1所述的方法，其特征在于，所述根据所述训练任务的目的，确定与所述训练任务匹配的数据采集任务，包括：
                                                                    ^^^^^^
                                                    逐字引用独权原文 + "包括："
```

### 权利要求布局顺序

标准顺序（所有模板一致）：

```
1. 方法独立权利要求
2-N. 方法从属权利要求（按技术流程的步骤顺序排列）
N+1. 装置/系统独立权利要求
N+2. 电子设备权利要求
N+3. 计算机可读存储介质权利要求
```

### 编号与标题格式

- 权利要求编号使用阿拉伯数字，格式为 `X. `（数字 + 英文句点 + 空格）
- 权利要求书标题为 `权  利  要  求  书`（每个字之间用两个空格隔开，符合中国专利局格式惯例）
- 标题使用黑体（SimHei）、三号字（32 half-points）、居中
- 正文使用宋体（SimSun）、小四号字（24 half-points）

## 算法、AI 与生物医学专利专项检查

撰写涉及算法、人工智能、生物医学或公共卫生决策的专利时，除通用检查外，还需逐项确认：

1. **数学模型可实施性**：关键变量、参数、目标函数、约束条件和优化过程均有可执行定义，避免仅写“建立模型”“进行优化”。
2. **输入输出闭环**：每个计算步骤明确输入数据、处理逻辑和输出结果；后续步骤引用的中间结果必须能由前序步骤得到。
3. **算法边界**：区分训练、推理、仿真、优化、控制执行等阶段；若未涉及训练，不应强行写成深度学习训练方案。
4. **医学用途边界**：涉及疾病、诊疗或公共卫生时，优先表述为“决策支持”“风险评估”“控制策略生成”，避免写成直接替代医生或行政机关作出诊疗/治理决定。
5. **技术效果可验证**：有益效果应落到计算稳定性、鲁棒性、模型适配效率、约束满足概率、策略可执行性、资源公平性等技术或工程指标。

## 智能制造与闭环控制专项规则

撰写智能制造、工业数据分析、产线检测、设备补偿控制类专利时，说明书和权利要求应形成稳定闭环，而不是只描述“智能分析”或“自动调整”。

### 系统架构必须说明数据流和控制流

系统架构至少说明以下角色中的相关部分：

- 检测设备：产生检测数据、图像数据、尺寸数据或设备状态；
- 电子设备/中控系统：执行绑定、筛除、计算、判断和指令生成；
- 数据库：存储绑定关系、基准数据、检测结果、执行结果和配置参数；
- 目标执行设备：接收补偿指令并执行微调、调整或控制动作；
- 看板/追溯系统：展示偏移趋势、补偿记录、异常提示和良率关联数据。

### 方法实施例必须形成闭环

优先按以下因果链组织说明书：

```
检测数据 → 绑定关系 → 基准数据 → 失真筛除 → 有效样本
→ 偏移计算 → 免调判断 → 补偿指令 → 设备执行 → 结果回写
```

每一环都要说明输入、处理和输出。若某一环不是本案必要特征，应说明其作为可选实施方式，而不是强行写入独立权利要求。

### 安全策略必须可执行

涉及自动补偿或设备微调时，说明书应写明至少一种安全策略：

- 免调区间：区分允许波动和需要补偿的系统性偏移；
- 最大补偿限幅：避免超过设备安全范围或工艺允许范围；
- 单次调整步长：避免过调、震荡或一次性大幅调整；
- 连续异常提示：识别治具磨损、载具定位异常、设备漂移或检测异常；
- 通讯/执行失败处理：失败时不视为已完成补偿，并回写异常状态。

### 追溯与看板不要只写商业效果

追溯看板应落到具体数据字段或统计维度，例如：

- 当前生产对象、目标补偿对象、目标装配设备、测量点位；
- 平均偏移量、补偿方向、补偿量、执行结果、检测结果；
- 按产品型号、工站、设备、治具、载具或时间窗口统计的趋势。

有益效果应对应技术特征，例如“失真筛除”对应“减少误补偿”，“免调区间+步长限制”对应“降低频繁调整和过调风险”，“回写追溯”对应“形成可验证的闭环记录”。

## 传染病动力学与公共卫生控制专项规则

撰写传染病动力学优化控制类专利时，权利要求和说明书应形成以下因果链：

1. **仓室模型信息**：说明仓室集合、状态变量、模型参数和微分方程表达式如何输入或生成。
2. **不确定性场景信息**：说明经验分布、参数样本、状态估计、Wasserstein 模糊集或其他场景集合如何构造。
3. **动力学仿真信息**：说明确定性 ODE 或随机 SDE 求解器如何基于场景、状态和控制动作得到预测轨迹。
4. **控制动作信息**：说明疫苗接种、非药物干预、隔离、治疗、资源调配或自定义参数缩放如何注入动力学方程。
5. **约束与目标信息**：说明健康成本、经济成本、公平性成本、预算约束、边界约束、机会约束如何参与优化。
6. **滚动控制信息**：若主张 MPC，应明确每个决策时刻只执行第一步控制动作，并在下一时刻基于新观测重新优化。
7. **延迟执行信息**：若主张政策延迟，应分别说明行政时间偏移 delay 和执行惯性 tau，而不是将延迟泛化为一个抽象词。

## 中国发明专利草案推荐结构

完整草案默认包括：

1. **摘要**：概括技术领域、核心步骤和主要技术效果。
2. **权利要求书**：优先采用方法、系统、电子设备、计算机可读存储介质的组合保护。
3. **说明书**：
   - 技术领域；
   - 背景技术；
   - 发明内容；
   - 附图说明；
   - 具体实施方式。
4. **附图建议**：至少包括系统架构图、方法流程图、滚动 MPC 闭环图、控制动作注入示意图。
